# WoofX3 UI Specification

## 1. Overview

WoofX3 UI is a multi-tenant SaaS control plane for managing live stream automation via the WoofX3 engine. The system provides authenticated users with a web interface to manage their streaming workflows, assets, modules, and overlay scenes across one or more engine instances.

### 1.1 Purpose

The platform enables streamers and broadcast professionals to:

- Create and manage automated workflows triggered by stream events (follows, subscriptions, bits, raids)
- Organize and serve media assets for use in overlays and alerts
- Install and manage modular extensions that extend the platform's capabilities
- Design and preview browser-based overlay scenes with layered widgets
- Monitor live stream status and receive real-time event feeds
- Collaborate with team members through shared account access

### 1.2 Scope

This document describes the functional specification for the WoofX3 UI control plane. It covers user-facing features, data models, system behavior, and integration patterns. The specification does not include implementation details, code-level architecture, or deployment procedures.

## 2. System Architecture

### 2.1 Multi-Tenant Model

The platform employs a hierarchical multi-tenant structure:

```
User → Account → Instance → Application
```

- **Users**: Individuals with authenticated access to the platform
- **Accounts**: Tenant containers representing organizations or individual streamers; the top-level billing and ownership unit
- **Instances**: Individual WoofX3 engine deployments, each associated with one account
- **Applications**: Engine-internal application contexts, identified by `applicationId`; each instance may contain one or more applications

### 2.2 Identity Management

The system distinguishes between two identity namespaces:

- **`instanceId`**: Generated by the platform (Convex); identifies an engine deployment; stored in both platform and engine databases
- **`applicationId`**: Generated by the engine; identifies an internal application context; returned to the platform during registration handshake and stored for routing purposes

Neither side generates the other's identifier.

### 2.3 Communication Architecture

The platform communicates with engine instances through two distinct channels:

**Channel 1: Platform to Engine (Server-Side Proxy)**
- Browser clients issue requests to the platform backend (Convex)
- The backend, acting as a trusted proxy, forwards requests to the appropriate engine instance
- All proxied requests include the engine's `applicationId` for correct scoping
- This channel handles sensitive operations requiring server-side execution

**Channel 2: Engine to Platform (Webhook Callbacks)**
- Engine instances POST to a shared platform webhook endpoint
- Callbacks include both `instanceId` (for tenant routing) and `applicationId` (for internal scoping)
- The endpoint validates `instanceId` before processing and uses upsert semantics to ensure idempotency
- Callbacks trigger reactive updates to connected browser clients

**Channel 3: Browser to Engine (Real-Time Transport)**
- Browser clients maintain direct WebSocket connections to engine instances for real-time data
- This channel handles live chat, stream events, and workflow execution status
- Reserved exclusively for real-time communication; never used for Convex-proxied calls

### 2.4 Account Sharing

Multiple users may belong to the same account, granting all members access to that account's instances and applications. Account sharing is managed entirely within the platform backend; the engine has no concept of users or accounts—it operates solely on `applicationId`.

## 3. Authentication and Authorization

### 3.1 Authentication Methods

The platform supports authentication via:

- **Twitch OAuth**: Users may sign in using their Twitch account through an OAuth flow managed by the platform backend
- **Password Authentication**: Traditional email/password registration and login through the platform's auth system

### 3.2 Session Management

Authenticated sessions are managed through JWT tokens issued by the platform's Convex backend. Sessions persist across browser sessions until explicitly signed out.

### 3.3 Route Guards

The application enforces authentication and onboarding status through route guards:

- **AuthGuard**: Redirects unauthenticated users to the login page
- **OnboardingGuard**: Requires users to have completed initial setup (account and instance creation) before accessing protected routes

### 3.4 Authorization Model

User permissions within an account are determined by role assignments:

- **Owner**: Full administrative control over the account and all instances
- **Admin**: Elevated permissions for account management; may manage instances and team members
- **Member**: Standard access to account resources within their permission level

## 4. User Interface

### 4.1 Application Shell

The main application interface consists of:

- **Header Bar**: Instance switcher dropdown, stream status indicator, quick actions, notifications, user menu
- **Navigation Dock**: Primary navigation links (Dashboard, Modules, Workflows, Assets, Scenes) and utility links (Team, Settings)
- **Main Content Area**: Dynamic content area for the active route
- **Command Palette**: Global keyboard-accessible action launcher (Cmd+K / Ctrl+K)

### 4.2 Design Principles

The interface follows a "Professional Control Plane" design philosophy:

- Aesthetic inspired by broadcast control systems (Premiere Pro, OBS Studio) combined with modern SaaS patterns
- Emphasis on information density, quick access, and spatial organization
- Clear visual hierarchy with consistent spacing, typography, and color usage
- Support for light and dark themes with customizable preset options

### 4.3 Responsive Behavior

The interface adapts to different screen sizes:

- Primary navigation collapses to icon-only mode on smaller screens
- Grid-based layouts reflow to accommodate available space
- Mobile-optimized views for critical monitoring functions

## 5. Feature Specifications

### 5.1 Dashboard

The dashboard provides a configurable workspace of resizable panels for real-time monitoring and quick actions.

**Available Modules:**

- **Chat Client**: Displays live chat messages from the stream
- **Workflow Runs**: Shows recent workflow execution history with status indicators
- **Event Feed**: Lists stream events (follows, subscriptions, raids, bits)
- **Macro Pad**: Quick-access buttons for triggering manual actions

**Behavior:**

- Users may add, remove, and reorder modules
- Module layouts persist per user per instance
- Layouts save automatically with debounced persistence
- Users may reset to a default layout
- Individual modules support fullscreen/maximize mode

**Data Sources:**

- Layout configuration stored in platform backend
- Live data (chat, events, workflow runs) sourced from engine via real-time transport

### 5.2 Workflows

Workflows automate actions in response to stream events. The platform supports both list management and visual workflow creation.

**List View (`/workflows`):**

- Display all workflows for the current instance
- Show workflow name, description, enabled status
- Actions: Open editor, duplicate, delete, toggle enabled
- Support filtering by enabled/disabled state
- Display workflow templates from the platform catalog

**Workflow Creation (`/workflows/new`):**

- Step-based creation wizard
- Preset templates for common Twitch events (follow, subscribe, bits, raid, gift)
- Trigger selection with configurable parameters
- Action selection with field configuration
- Preview of workflow structure before saving

**Visual Workflow Builder (`/workflows/:id`):**

- Canvas-based graph editor using React Flow
- Node types: Triggers, Actions, Conditions, Delays
- Node library sidebar with categorized components
- Visual connection lines between nodes (Bezier curves)
- Minimap for navigation in large workflows
- Zoom and pan controls
- Save/publish workflow state

**Workflow Node Types:**

- **Triggers**: Events that initiate workflow execution (chat commands, stream events, scheduled timers)
- **Actions**: Operations performed in response (send message, play sound, trigger OBS, call API)
- **Conditions**: Branch logic based on variable evaluation
- **Delays**: Pause execution for specified duration

**Data Model:**

- Workflows are stored in the platform backend with visual editor state (nodes and edges)
- Workflows may be synced to the engine for execution
- Engine execution status tracked via real-time transport

### 5.3 Assets

Asset management provides media file storage and organization for use in overlays and workflows.

**Capabilities:**

- Upload files via drag-and-drop or file picker
- Supported types: Images, Audio, Video
- Organize files into virtual folder hierarchy
- View assets in grid or table layout
- Search assets by name
- Filter by file type
- Sort by name, size, or date
- Download, rename, and delete assets
- Bulk selection and actions

**Storage:**

- Files stored through configurable storage backends (platform native, R2, or local)
- Metadata stored in platform backend
- Signed upload URLs generated server-side for direct browser upload

### 5.4 Modules

Modules extend the platform's functionality through a plugin system.

**Module Catalog:**

- Browse available modules by category
- Search modules by name or description
- View module details: version, author, description, triggers, actions
- See installation status (installed/not installed)

**Module Installation:**

- Install modules from the catalog to the current instance
- Upload custom module packages (ZIP format)
- Inspect module manifest and file structure
- Validate module before installation
- View installation status and errors

**Module Management:**

- Enable or disable installed modules
- Uninstall modules from the instance
- Module lifecycle managed on the engine
- Platform catalogs module metadata; engine tracks runtime state

### 5.5 Scenes

Scenes define layered visual layouts for browser-based overlay rendering in OBS or streaming software.

**Scene List (`/scenes`):**

- View all scenes for the current instance
- Create new scenes with configurable dimensions (default: 1920x1080)
- Edit, duplicate, or delete scenes
- Set default scene for browser source

**Scene Editor (`/scenes/:id`):**

- Canvas-based visual editor representing the overlay layout
- Widget library for adding elements (text, images, shapes, alerts)
- Layer management (z-index ordering)
- Alignment and spacing tools
- Drag-and-drop positioning
- Resize handles for widgets
- Undo/redo support
- Real-time preview

**Widget Types:**

- **Text**: Static or dynamic text with customizable font, size, color
- **Image**: Display images with optional animations
- **Shape**: Rectangles, circles, lines with styling
- **Alert Layers**: Configurable alert presentations linked to alert descriptors

**Scene Slots:**

- Scenes may define named slots for dynamic content
- Slots have position, size, and queue mode (stack, concurrent, interrupt)
- Multiple alerts can queue in a slot based on queue mode

### 5.6 Browser Source Overlay

The browser source overlay renders configured scenes for use in OBS or other streaming software.

**Rendering Behavior:**

- Browser source connects to a unique source key
- Claims the scene configuration and alert descriptors
- Polls for pending alerts
- Renders alert layers with configured animations
- Updates alert state as they complete

**Alert Lifecycle:**

1. Engine sends alert webhook to platform
2. Platform queues alert with state "pending"
3. Browser source polls and claims alert (state "rendering")
4. Browser source renders alert animation
5. Browser source updates alert state to "complete"
6. Alert moved to history

**Alert Descriptors:**

- Configure alert behavior per scene slot
- Support for multiple alert types (follow, subscription, bits, etc.)
- Priority and TTL (time-to-live) settings
- Layer configuration: text, image, video, audio, Lottie animations
- Animation in/out presets
- Hook support for custom JavaScript execution

### 5.7 Team Management

Account owners and admins may invite members and manage their roles.

**Member Management:**

- View current team members
- See member roles and status (active, invited, inactive)
- Invite new members by email
- Assign roles: Admin, Member
- Revoke invitations
- Remove members from account

**Invitation Flow:**

- Owner/admin sends invitation with selected role
- Invitee receives email with acceptance link
- Invitation expires after configurable duration
- Accepted invitations grant appropriate access

### 5.8 Settings

Settings pages allow users to configure their account, profile, and instance preferences.

**Account Settings:**

- Account name and display preferences
- Billing information (placeholder for payment integration)

**Profile Settings:**

- Display name and avatar
- Email preferences
- Notification settings

**Appearance Settings:**

- Theme selection (light, dark, system)
- Theme presets with color customization

**Engine Settings:**

- View and update engine URL
- Test connection to engine
- Manage platform links (Twitch OAuth)

### 5.9 Stream Status

The header displays current stream status when the engine is connected.

**Status Indicators:**

- Live/Offline indicator with animated pulse when live
- Stream uptime timer
- Viewer count (when live)

**Data Source:**

- Stream status sourced from engine via real-time transport
- Polling interval: 5 seconds

## 7. API Surface

### 7.1 Platform Backend API (Convex)

The platform backend exposes:

**Queries:**
- Fetch account information
- List instances for current user
- Query workflow definitions
- Retrieve module catalog
- List assets and folders
- Query scene configurations
- Fetch dashboard layouts
- Get platform links

**Mutations:**
- Create/update/delete accounts
- Create/update/delete instances
- Manage account members and invitations
- CRUD workflows
- CRUD assets and folders
- CRUD scenes and scene slots
- Configure alert descriptors
- Save dashboard layouts
- Manage chat commands
- Update platform links

**Actions:**
- Sync workflows to engine
- Test engine connection
- Upload module packages
- Generate signed upload URLs

**Internal Functions:**
- Handle engine webhook callbacks (alert ingestion, module lifecycle)
- Process Twitch OAuth flows
- Validate registration handshakes

### 7.2 HTTP Endpoints

**Authentication:**

- `GET /api/auth/twitch/start` - Initiate Twitch OAuth flow
- `GET /api/auth/twitch/callback` - Handle Twitch OAuth callback
- Standard Convex auth routes for password authentication

**Webhooks:**

- `POST /api/webhooks/woofx3/alerts` - Receive alerts from engine
- `POST /api/webhooks/woofx3/notify` - Receive notifications from engine (with webhook secret validation)

**Browser Source:**

- `POST /api/browser-source/:key/claim` - Claim browser source with key
- `PATCH /api/browser-source/alerts/:id` - Update alert state

**OBS Integration:**

- `GET /api/obs/commands?sceneId=` - Poll pending OBS commands
- `PATCH /api/obs/commands/:id` - Update command state

**Widget Assets:**

- `GET /api/widgets/:moduleId/:directory/*` - Serve module widget files

### 7.3 Real-Time Transport

The WoofxTransport interface provides:

**Connection Management:**
- Connect/disconnect to engine instance
- Connection status check

**Stream Operations:**
- Get stream status (one-off)
- Subscribe to chat messages (real-time)
- Subscribe to stream events (real-time)

**Workflow Operations:**
- Get workflows
- Create/update/delete workflows
- Execute workflow manually
- Subscribe to workflow runs (real-time)

**Module Operations:**
- Get module state
- List engine modules
- Uninstall module
- Set module state

## 8. Security Considerations

### 8.1 Authentication

- All user-facing routes require authentication
- Session tokens managed server-side
- OAuth flows use PKCE and state validation

### 8.2 Webhook Security

- Engine webhook callbacks validated against per-instance secrets
- Instance ID routing ensures data isolation
- Idempotent processing prevents duplicate operations

### 8.3 Authorization

- Instance-level access control ensures users only access permitted resources
- Role-based permissions for account management
- Engine operations scoped by application ID

### 8.4 Data Isolation

- Multi-tenant data separated by instance ID
- Storage backends isolated per deployment
- No cross-tenant data access

## 9. Integration Patterns

### 9.1 Engine Registration Handshake

1. User provides engine URL during onboarding
2. Platform creates instance record, generating instance ID
3. Platform POSTs to engine's `/api/register` with instance ID and webhook configuration
4. Engine stores registration data and returns application ID
5. Platform stores application ID for future requests

### 9.2 Alert Flow

1. Stream event occurs (follow, subscription, etc.)
2. Engine POSTs alert to platform webhook endpoint
3. Platform creates alert record in pending state
4. Browser source polls claims alert (state: rendering)
5. Browser source renders overlay animation
6. Browser source updates alert state to complete
7. Alert moved to history

### 9.3 Workflow Sync

1. User creates/updates workflow in visual editor
2. Workflow saved to platform backend
3. User triggers sync to engine
4. Platform backend builds engine-compatible payload from visual editor state
5. Platform backend calls engine API to create/update workflow
6. Engine returns workflow ID
7. Platform backend stores engine workflow ID for reference

## 15. Glossary

- **Alert**: A notification triggered by a stream event, rendered in a browser overlay
- **Application**: An engine-internal context for multi-tenancy; identified by applicationId
- **Dashboard Module**: A configurable panel component for the dashboard workspace
- **Engine**: The WoofX3 runtime engine that executes workflows and processes events
- **Instance**: A deployment of the WoofX3 engine, managed through this platform
- **Module**: A plugin that extends the platform's triggers, actions, or widgets
- **Scene**: A visual layout for browser-based overlays, composed of widgets and slots
- **Slot**: A named region within a scene for dynamic alert content
- **Widget**: A visual element (text, image, shape) placed on a scene canvas
- **Workflow**: An automation chain of triggers, conditions, and actions
